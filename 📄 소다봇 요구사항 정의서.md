# 📄 소다봇 요구사항 정의서 (최신)

- 문서 버전: v2.0
- 최종 갱신일: 2026-04-13
- 기준 코드: 현재 워크스페이스 구현

---

## 1. 개요

- 프로젝트명: 소다봇 (SodaBot)
- 목적: 디스코드 서버 운영 자동화(경고/인증/음성채널) 제공
- 실행 형태: 단일 봇 프로세스가 여러 서버(길드)를 동시에 처리

### 주요 기능

- 경고 부여/회수/조회
- 정기 자동 경고(AUTO_ADD)
- 경고 누적 기반 자동 강퇴(kick)
- 재입장 시 경고/인증 상태 복구
- 인증 만료 3일 전 안내
- 기간 전환 시 인증 완료 역할 초기화
- 현재 기간 인증 완료 상태 DB 저장 및 역할 복구
- 임시 개인 음성 채널 생성 및 자동 삭제
- `/설정` 슬래시 명령으로 런타임 설정 변경
- 관리자 테스트 명령(`/설정 테스트자동경고`, `/설정 테스트안내`)

---

## 2. 시스템 아키텍처

```text
Discord Gateway
  -> SodaBot (discord.py)
      -> SQLite (aiosqlite)
      -> In-memory config cache (guild scoped)
      -> APScheduler (KST, minute tick)
```

### 기술 스택

- Language: Python 3.12
- Framework: discord.py
- DB: SQLite (aiosqlite)
- Scheduler: APScheduler (AsyncIOScheduler)
- Timezone: KST(Asia/Seoul) 고정

### 멀티 서버(길드) 정책

- 모든 데이터/설정은 길드 단위로 분리되어야 함
- DB 테이블에 `guild_id`를 저장해야 함
- 설정 키는 `g:<guild_id>:<key>` 스코프로 저장해야 함
- 스케줄러는 모든 길드를 순회하며 길드별 로직을 독립 수행해야 함

## 3. 경고 기능 요구사항

### 3.1 경고 부여

- 명령어: `/경고 부여 [유저] [사유]`
- 권한: 관리자만 가능
- 서버(길드) 컨텍스트에서만 실행 가능

동작 순서:
1. 관리자 권한 확인
2. 사용자 현재 경고 수를 DB에서 조회
3. `warning_count`를 1 증가해 DB 저장
4. `warning_history`에 `ADD` 이력 저장
5. 경고 역할 동기화
6. 경고 로그 채널에 임베드 출력
7. 증가 후 경고 수가 3 이상이면 kick 시도

### 3.2 경고 회수

- 명령어: `/경고 회수 [유저] [사유]`
- 권한: 관리자만 가능
- 서버(길드) 컨텍스트에서만 실행 가능

동작 순서:
1. 관리자 권한 확인
2. `warning_count`를 1 감소(최저 0)
3. `warning_history`에 `REMOVE` 이력 저장
4. 경고 역할 동기화
5. 경고 로그 채널에 임베드 출력

정책:
- 경고 이력은 삭제하지 않고 누적 보존
- 현재 상태는 `users.warning_count`로 관리

### 3.3 경고 조회

- 명령어: `/경고 조회`, `/경고 조회 [유저]`
- 본인 조회: 누구나 가능
- 타인 조회: 관리자만 가능
- 출력: ephemeral 메시지

제공 정보:
- 현재 경고 횟수
- 최근 이력(액션, 처리자, 사유, 처리 시각, 처리 후 경고 수)

### 3.4 자동 강퇴(kick)

- 트리거: 경고 누적 수(`warning_count`)가 3 이상
- 핵심 정책: 강퇴 판단은 역할이 아니라 DB 경고 수 기준으로 동작해야 함
- kick 성공 시에만 `warning_history`에 `KICK` 이력을 저장해야 함

### 3.5 재입장 처리

- 멤버가 서버 재입장하면 최근 이력을 기준으로 처리해야 함

정책:
- 최근 이력이 `KICK`이면 `warning_count`를 0으로 초기화하고 `RESET` 이력을 저장해야 함
- 자진 퇴장 후 재입장이면 경고를 초기화하지 않아야 함
- 자진 퇴장 후 재입장 시 기존 `warning_count`에 맞는 경고 역할을 자동 재부여해야 함
- 기존 `ADD`/`AUTO_ADD`/`KICK` 이력은 보존해야 함

## 4. 자동 경고/안내 스케줄 요구사항

### 4.1 실행 주기

- 스케줄러는 매분 tick 실행
- 실제 동작은 KST 기준 특정 시점(00:00)에만 수행

### 4.2 인증 기간 규칙

- 일반 월:
  - 1차: 1일 ~ 15일
  - 2차: 16일 ~ 말일
- 2월:
  - 1차: 1일 ~ 14일
  - 2차: 15일 ~ 말일
- period key 형식: `YYYY-MM-1`, `YYYY-MM-2`

### 4.3 자동 경고 실행 시점

- KST 00:00 기준 아래 날짜에 실행
  - 매월 1일
  - 매월 16일
  - 2월 15일

### 4.4 자동 경고 대상 조건

- 멤버가 봇 계정이 아니어야 함
- 서버 접속 후 7일 이상 지난 멤버여야 함
- `auth_completed_role_id` 역할이 없어야 함
- `acquaintance_role_id` 역할이 없어야 함
- `bot_role_id` 역할이 없어야 함
- 같은 `period_key`에서 이미 AUTO_ADD를 받은 적이 없어야 함

### 4.5 자동 경고 안전 장치

- `auth_completed_role_id` 미설정/미존재 시 자동 경고를 스킵해야 함
- `warning_channel_id`가 미설정/미존재여도 자동 경고 자체는 계속 수행하고 로그만 생략 가능해야 함

### 4.6 안내 메시지

- 인증 종료 3일 전 KST 00:00에 인증 채널로 안내 전송
- 메시지 형식: `@everyone` 멘션 포함

안내일 규칙:
- 일반 월: 12일, (말일-3)
- 2월: 11일, (말일-3)

### 4.7 인증 완료 역할 초기화

- 자동 경고 처리 후, 인증 완료 역할을 가진 멤버에게서 해당 역할 제거
- 순서: `검사 -> 자동 경고 -> 인증 역할 제거`

### 4.8 인증 완료 상태 저장/복구

- 관리자가 인증 완료 역할을 부여하면 현재 기간 인증 완료 상태를 DB에 저장해야 함
- 현재 기간 인증 완료 역할이 제거되면 현재 기간 인증 상태도 DB에서 미완료로 갱신해야 함
- 멤버가 재입장했을 때 현재 기간 인증 완료 상태가 DB에 남아 있으면 인증 완료 역할을 자동 재부여해야 함
- 봇 재시작 시 현재 인증 완료 역할을 가진 멤버는 현재 기간 인증 완료 상태로 DB에 동기화해야 함

## 5. 음성 채널 기능 요구사항

### 5.1 개인 임시 채널 생성

- 트리거: 사용자가 `voice_create.trigger_channel_id` 채널에 입장
- 전제: `voice_create.enabled == true`
- 생성 위치: `voice_create.category_id` 카테고리 하위
- 생성 후 대상 사용자를 새 채널로 이동

생성 규칙:
- 채널명은 사용자 닉네임 기반(허용 문자 sanitize 적용)
- 사용자 ID를 소유자 기준으로 DB(`voice_channels`)에 기록
- 기존 채널 존재 여부와 무관하게 매번 새 채널 생성

권한 규칙:
- 생성 채널에서 소유자에게 채널 관리/이동 관련 권한 부여

### 5.2 자동 삭제

- 전제: `voice_create.auto_delete == true`
- 소유권이 기록된 임시 채널에서 non-bot 멤버 수가 0이면 자동 삭제
- 삭제 후 소유권 매핑(`voice_channels`)도 함께 정리

## 6. 설정 관리 요구사항

### 6.1 설정 저장/조회

- 설정은 DB에 영속 저장, 메모리 캐시에 적재
- 변경 시 DB와 캐시를 즉시 동기화
- 봇 재시작 없이 즉시 반영

### 6.2 관리자 설정 명령

- `/설정 경고채널 [채널]`
- `/설정 인증채널 [채널]`
- `/설정 인증완료역할 [역할]`
- `/설정 지인역할 [역할]`
- `/설정 봇역할 [역할]`
- `/설정 경고역할1 [역할]`
- `/설정 경고역할2 [역할]`
- `/설정 경고역할3 [역할]`
- `/설정 음성채널 활성화 [ON/OFF]`
- `/설정 음성채널 트리거 [채널]`
- `/설정 음성채널 카테고리 [카테고리]`
- `/설정 음성채널 자동삭제 [ON/OFF]`

### 6.3 관리자 테스트 명령

- `/설정 테스트자동경고`
  - 현재 길드 대상 자동 경고 로직 즉시 실행
- `/설정 테스트안내`
  - 현재 길드 대상 인증 안내 메시지 즉시 전송

### 6.4 필수 설정 검증

- 시작 시 길드별 다음 키 미설정이면 경고 로그 출력
  - `warning_channel_id`
  - `auth_channel_id`
  - `auth_completed_role_id`

## 7. 데이터 구조

### 7.1 users

```text
users
- id (PK)
- discord_id (TEXT, UNIQUE, 실제 저장값: "<guild_id>:<user_id>")
- guild_id (TEXT)
- warning_count (INTEGER)
- created_at (TEXT, KST 포맷)
- updated_at (TEXT, KST 포맷)
```

### 7.2 warning_history

```text
warning_history
- id (PK)
- user_id (FK -> users.id)
- guild_id (TEXT)
- action (ADD / REMOVE / AUTO_ADD / KICK / RESET)
- actor_discord_id (TEXT, nullable)
- target_discord_id (TEXT)
- reason (TEXT)
- total_warning_count (INTEGER)
- period_key (TEXT, nullable)
- created_at (TEXT, KST 포맷)
```

### 7.3 config

```text
config
- id (PK)
- key (TEXT, UNIQUE, 실제 저장값: g:<guild_id>:<key>)
- guild_id (TEXT)
- value (TEXT)
- value_type (TEXT: bool/int/str)
- updated_by_discord_id (TEXT, nullable)
- updated_at (TEXT, KST 포맷)
```

### 7.4 voice_channels

```text
voice_channels
- id (PK)
- channel_id (TEXT, UNIQUE)
- guild_id (TEXT)
- owner_discord_id (TEXT)
- created_at (TEXT, KST 포맷)
```

### 7.5 auth_period_status

```text
auth_period_status
- id (PK)
- guild_id (TEXT)
- user_discord_id (TEXT)
- period_key (TEXT)
- is_authenticated (INTEGER: 0/1)
- updated_by_discord_id (TEXT, nullable)
- created_at (TEXT, KST 포맷)
- updated_at (TEXT, KST 포맷)
```

## 8. 비기능 요구사항

- 모든 권한 판단은 관리자 권한 기반으로 수행
- 모든 시간 계산/저장은 KST 기준
- 슬래시 명령은 DM이 아닌 길드에서만 동작해야 하는 기능을 명확히 제한
- 오류 발생 시 서비스 전체 중단 대신 길드 단위로 격리해 처리 지속
- 시작/종료 안정성 보장
  - 종료 시 스케줄러와 DB 연결을 정리
  - 네트워크/게이트웨이 오류 시 재시도 루프 적용
- 식별은 이름이 아닌 ID 기반으로 처리

## 9. 구현 주의사항 및 현행 차이

- `voice_create.allow_duplicate` 키는 기본 설정에 존재하지만, 현재 구현은 값과 무관하게 중복 생성 허용(항상 새 채널 생성) 정책으로 동작함
- 강퇴는 경고 역할 부여 여부와 독립적으로, DB 경고 누적값 기준으로 판단함
- 경고 역할은 상태 표시/동기화 목적이며 강퇴의 근거 데이터는 아님
