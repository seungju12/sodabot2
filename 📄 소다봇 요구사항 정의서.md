# 📄 소다봇 요구사항 정의서 (최신)

- 문서 버전: v3.0
- 최종 갱신일: 2026-04-30
- 기준 코드: 현재 워크스페이스 구현

---

## 1. 개요

- 프로젝트명: 소다봇 (SodaBot)
- 목적: 디스코드 서버 운영 자동화(경고/인증/음성채널) 제공
- 실행 형태: 단일 봇 프로세스가 여러 서버(길드)를 동시에 처리

### 주요 기능

- 경고 부여/회수/조회
- 신규 입장자 버튼 기반 초기 설정 후 역할 부여
- 정기 자동 경고(AUTO_ADD)
- 경고 누적 기반 자동 강퇴(kick)
- 재입장 시 경고/인증 상태 복구
- 인증 채널을 통한 인증 기간 안내
- 권한받기 채널을 통한 신규 입장 초기 설정
- 기간 전환 시 인증 완료 역할 초기화
- 현재 기간 인증 완료 상태 DB 저장 및 역할 복구
- 임시 개인 음성 채널 생성 및 자동 삭제
- `/설정` 슬래시 명령으로 런타임 설정 변경
- 관리자 테스트 명령(`/설정 테스트자동경고`, `/설정 테스트안내`)

## 2. 시스템 아키텍처

```text
Discord Gateway
  -> SodaBot (discord.py)
      -> PostgreSQL
      -> In-memory config cache (guild scoped)
      -> APScheduler (KST, minute tick)
```

### 기술 스택

- Language: Python 3.12
- Framework: discord.py
- DB: PostgreSQL
- Scheduler: APScheduler (AsyncIOScheduler)
- Timezone: KST(Asia/Seoul) 고정

### 멀티 서버(길드) 정책

- 모든 데이터/설정은 길드 단위로 분리되어야 함
- DB 테이블에 `guild_id`를 저장해야 함
- 설정 키는 `g:<guild_id>:<key>` 스코프로 저장해야 함
- 스케줄러는 모든 길드를 순회하며 길드별 로직을 독립 수행해야 함

### 신규 입장 초기 설정 요구사항

목표:
- 신입 멤버는 권한받기 채널에서 초기 설정을 완료하기 전까지 남성/여성/지인 역할 없이 제한된 상태여야 함
- 기존의 단순 이모지 반응 역할 부여 방식은 단계적으로 대체하고, 임베드 + 버튼 기반 상호작용으로 통합해야 함

필수 입력 항목:
- 성별 1개 선택
  - 남성 또는 여성
- 자주 하는 게임 1개 선택
  - 롤, 오버워치, 배틀그라운드, 기타 중 1개

동작 정책:
1. 신입이 입장하면 권한받기 채널에 고정된 초기 설정 안내 임베드를 확인할 수 있어야 함
   - 이 채널은 인증 기간 공지용 인증채널과 별도로 운영되어야 함
2. 임베드에는 역할 받기 전, 서버에서 정한 닉네임 양식대로 먼저 변경하라는 안내가 명확히 포함되어야 함
3. 일반 입장 버튼을 누를 때 현재 서버 닉네임이 아래 형식을 만족하지 않으면 다음 단계로 진행되면 안 됨
  - 형식: `두 자리 년생 + 공백 + 닉네임`
  - 예시: `01 단풍소다`
  - 닉네임 부분 길이: 2~4글자
  - 형식 불일치 시 ephemeral로 수정 안내를 제공해야 함
4. 지인 입장 버튼을 누를 때 현재 서버 닉네임이 아래 형식을 만족하지 않으면 역할 지급이 진행되면 안 됨
  - 형식: `닉네임_OO지인`
  - 예시: `소다_단풍소다지인`
  - 앞 닉네임 부분 길이: 2~4글자
  - 뒤 닉네임 부분 길이: 2~4글자
  - 형식 불일치 시 ephemeral로 수정 안내를 제공해야 함
5. 일반 신입은 임베드의 버튼을 통해 성별 1개와 자주 하는 게임 1개를 선택해야 함
6. 일반 입장 버튼을 누르면 먼저 성별 선택 화면으로 전환되어야 함
7. 성별을 선택하면 별도 완료 버튼 없이 즉시 게임 선택 화면으로 넘어가야 함
8. 게임을 선택하면 별도 최종 완료 버튼 없이 즉시 아래 역할을 반영해야 함
  - 성별 역할 1개
  - 게임 역할 1개
9. 이 역할들이 서버 채널 접근 기준 역할로 사용될 수 있어야 함
10. 지인으로 들어온 사람은 일반 절차를 거치지 않고, 성별/게임 선택 없이 지인 역할만 부여할 수 있어야 함
11. 사용자가 잘못 선택한 경우 다시 진행할 수 있어야 하며, 같은 분류의 기존 선택은 새 선택으로 대체되어야 함
12. 완료 여부를 별도 DB 상태로 강제 저장하지 않아도 되며, 재진행 가능성이 항상 열려 있어야 함

UX 정책:
- 공개 채널에는 안내용 임베드만 유지하고, 실제 진행 결과/에러 안내는 가능한 한 ephemeral 응답으로 처리해야 함
- 사용자는 현재 선택 상태와 남은 단계(예: 1/2 완료)를 명확히 확인할 수 있어야 함
- 완료 전에는 왜 권한이 부여되지 않았는지 안내 문구가 분명해야 함
- 잘못 선택했을 때 사용할 수 있는 처음으로 또는 다시 진행 동선을 직관적으로 제공해야 함
- 안내 임베드에는 일반 입장 예시와 지인 입장 예시를 서로 혼동하지 않도록 분리해 표시해야 함
- 안내 임베드는 과하지 않은 이모지를 포함해 단계가 한눈에 보이는 형태로 구성할 수 있어야 함
- 진행 임베드의 선택 현황에는 이미 고른 항목만 표시하고, 미선택 항목은 장황하게 노출하지 않는 편이 적합함

예외 처리:
- 설정 대상 역할이나 채널이 삭제/미설정이면 관리자가 즉시 인지할 수 있도록 로그를 남겨야 함
- 권한 부여 실패 시 사용자에게 재시도 안내를 제공해야 함
- 관리자는 필요 시 특정 멤버의 역할 받기 절차를 다시 안내할 수 있어야 함

## 3. 경고 기능 요구사항

### 3.1 경고 부여

- 명령어: `/경고 부여 [유저] [사유]`
- 권한: 관리자만 가능
- 서버(길드) 컨텍스트에서만 실행 가능

동작 순서:
1. 관리자 권한 확인
2. 사용자 현재 경고 수를 DB에서 조회
3. `warning_count`를 1 증가해 DB 저장
4. `warning_history`에 `ADD` 이력 저장
5. 경고 역할 동기화
6. 경고 로그 채널에 임베드 출력
7. 증가 후 경고 수가 3 이상이면 kick 시도

### 3.2 경고 회수

- 명령어: `/경고 회수 [유저] [사유]`
- 권한: 관리자만 가능
- 서버(길드) 컨텍스트에서만 실행 가능

동작 순서:
1. 관리자 권한 확인
2. `warning_count`를 1 감소(최저 0)
3. `warning_history`에 `REMOVE` 이력 저장
4. 경고 역할 동기화
5. 경고 로그 채널에 임베드 출력

정책:
- 경고 이력은 삭제하지 않고 누적 보존
- 현재 상태는 `users.warning_count`로 관리

### 3.3 경고 조회

- 명령어: `/경고 조회`, `/경고 조회 [유저]`
- 본인 조회: 누구나 가능
- 타인 조회: 관리자만 가능
- 출력: ephemeral 메시지

제공 정보:
- 현재 경고 횟수
- 최근 이력(액션, 처리자, 사유, 처리 시각, 처리 후 경고 수)

### 3.4 자동 강퇴(kick)

- 트리거: 경고 누적 수(`warning_count`)가 3 이상
- 핵심 정책: 강퇴 판단은 역할이 아니라 DB 경고 수 기준으로 동작해야 함
- kick 성공 시에만 `warning_history`에 `KICK` 이력을 저장해야 함

### 3.5 재입장 처리

- 멤버가 서버 재입장하면 최근 이력을 기준으로 처리해야 함

정책:
- 최근 이력이 `KICK`이면 `warning_count`를 0으로 초기화하고 `RESET` 이력을 저장해야 함
- 자진 퇴장 후 재입장이면 경고를 초기화하지 않아야 함
- 자진 퇴장 후 재입장 시 기존 `warning_count`에 맞는 경고 역할을 자동 재부여해야 함
- 기존 `ADD`/`AUTO_ADD`/`KICK` 이력은 보존해야 함

## 4. 자동 경고/안내 스케줄 요구사항

### 4.1 실행 주기

- 스케줄러는 매분 tick 실행
- 실제 동작은 KST 기준 특정 시점(00:00)에만 수행

### 4.2 인증 기간 규칙

- 일반 월:
  - 1차: 1일 ~ 15일
  - 2차: 16일 ~ 말일
- 2월:
  - 1차: 1일 ~ 14일
  - 2차: 15일 ~ 말일
- period key 형식: `YYYY-MM-1`, `YYYY-MM-2`

### 4.3 자동 경고 실행 시점

- KST 00:00 기준 아래 날짜에 실행
  - 매월 1일
  - 매월 16일
  - 2월 15일

### 4.4 자동 경고 대상 조건

- 멤버가 봇 계정이 아니어야 함
- 서버 접속 후 7일 이상 지난 멤버여야 함
- `auth_completed_role_id` 역할이 없어야 함
- `acquaintance_role_id` 역할이 없어야 함
- `bot_role_id` 역할이 없어야 함
- 같은 `period_key`에서 이미 AUTO_ADD를 받은 적이 없어야 함

### 4.5 자동 경고 안전 장치

- `auth_completed_role_id` 미설정/미존재 시 자동 경고를 스킵해야 함
- `warning_channel_id`가 미설정/미존재여도 자동 경고 자체는 계속 수행하고 로그만 생략 가능해야 함
- 자동 경고 스케줄링 종료 후 경고 채널에 결과 요약 리스트를 남겨야 함
  - 경고 1회
  - 경고 2회
  - 경고 3회(추방자)

### 4.6 안내 메시지

- 인증 종료 3일 전 KST 00:00에 인증 채널로 안내 전송
- 메시지 형식: `@everyone` 멘션 포함

안내일 규칙:
- 일반 월: 12일, (말일-3)
- 2월: 11일, (말일-3)

### 4.7 인증 완료 역할 초기화

- 자동 경고 처리 후, 인증 완료 역할을 가진 멤버에게서 해당 역할 제거
- 순서: `검사 -> 자동 경고 -> 인증 역할 제거`

### 4.8 인증 완료 상태 저장/복구

- 관리자가 인증 완료 역할을 부여하면 현재 기간 인증 완료 상태를 DB에 저장해야 함
- 현재 기간 인증 완료 역할이 제거되면 현재 기간 인증 상태도 DB에서 미완료로 갱신해야 함
- 멤버가 재입장했을 때 현재 기간 인증 완료 상태가 DB에 남아 있으면 인증 완료 역할을 자동 재부여해야 함
- 봇 재시작 시 현재 인증 완료 역할을 가진 멤버는 현재 기간 인증 완료 상태로 DB에 동기화해야 함
- 연속된 2개 기간에 인증한 멤버는 자동 경고 스케줄링이 끝난 뒤 경고 1회를 자동 회수해야 함
- 보상은 겹치는 연속 구간마다 중복 적용되면 안 됨
  - 예: 2회 연속 인증 시 1회 차감, 3회 연속에서는 추가 차감 없음, 4회 연속에서 다시 1회 차감
- 같은 기간에 같은 연속 인증 보상이 중복 적용되면 안 됨

## 5. 음성 채널 기능 요구사항

### 5.1 개인 임시 채널 생성

- 트리거: 사용자가 `voice_create.trigger_channel_id` 채널에 입장
- 전제: `voice_create.enabled == true`
- 생성 위치: `voice_create.category_id` 카테고리 하위
- 생성 후 대상 사용자를 새 채널로 이동

생성 규칙:
- 채널명은 사용자 닉네임 기반(허용 문자 sanitize 적용)
- 사용자 ID를 소유자 기준으로 DB(`voice_channels`)에 기록
- 기존 채널 존재 여부와 무관하게 매번 새 채널 생성
- 기존에 생성된 개인 임시 음성 채널이 남아 있어도 재사용하지 않음

권한 규칙:
- 생성 채널에서 소유자에게 채널 관리/이동 관련 권한 부여

### 5.2 자동 삭제

- 전제: `voice_create.auto_delete == true`
- 소유권이 기록된 임시 채널에서 non-bot 멤버 수가 0이면 자동 삭제
- 삭제 후 소유권 매핑(`voice_channels`)도 함께 정리

## 6. 설정 관리 요구사항

### 6.1 설정 저장/조회

- 설정은 DB에 영속 저장, 메모리 캐시에 적재
- 변경 시 DB와 캐시를 즉시 동기화
- 인증채널과 권한받기 채널은 별도 설정값으로 관리해야 함
- 봇 재시작 없이 즉시 반영

### 6.2 관리자 설정 명령

- `/설정 경고채널 [채널]`
- `/설정 인증채널 [채널]` → 인증 기간 공지 및 안내 메시지 전송용 채널
- `/설정 인증스케줄링 [ON/OFF]` → 자동 인증 안내 및 자동 경고 스케줄링 활성화 여부
- `/설정 권한받기채널 [채널]` → 신규 입장자의 초기 설정 진행용 채널
- `/설정 인증완료역할 [역할]` → 기간 인증 완료 상태를 나타내는 역할
- `/설정 남성역할 [역할]`
- `/설정 여성역할 [역할]`
- `/설정 롤역할 [역할]`
- `/설정 오버워치역할 [역할]`
- `/설정 배그역할 [역할]`
- `/설정 기타게임역할 [역할]`
- `/설정 권한안내전송`
- `/설정 지인역할 [역할]`
- `/설정 봇역할 [역할]`
- `/설정 경고역할1 [역할]`
- `/설정 경고역할2 [역할]`
- `/설정 경고역할3 [역할]`
- `/설정 음성채널 활성화 [ON/OFF]`
- `/설정 음성채널 트리거 [채널]`
- `/설정 음성채널 카테고리 [카테고리]`
- `/설정 음성채널 자동삭제 [ON/OFF]`

추가 정책:
- 자동 인증 안내와 자동 경고 스케줄링은 길드별 ON/OFF 토글이 가능해야 함
- 스케줄링 OFF 상태에서는 정기 tick 기반 자동 인증 안내와 자동 경고를 실행하지 않아야 함
- 스케줄링 OFF 상태여도 관리자 테스트 명령으로 수동 실행은 가능해야 함
- 초기 설정용 성별/게임 역할은 고정 선택지에 맞는 개별 설정값으로 관리되어야 함
- 지인 입장자를 위한 지인 역할은 별도 설정값으로 유지해야 함
- 권한받기 채널에는 관리자 명령으로 초기 설정 안내 임베드를 재전송할 수 있어야 함
- 인증채널의 안내/공지 메시지와 권한받기 채널의 온보딩 메시지는 서로 독립적으로 관리되어야 함
- 닉네임 검사는 현재 고정 정책으로 운영한다
  - `두 자리 년생 + 공백 + 2~4글자 닉네임`
  - 지인 입장은 별도 고정 정책으로 운영한다
  - `2~4글자 닉네임 + _ + 2~4글자 닉네임 + 지인`

### 6.3 관리자 테스트 명령

- `/설정 테스트자동경고`
  - 현재 길드 대상 자동 경고 로직 즉시 실행
- `/설정 테스트안내`
  - 현재 길드 대상 인증 안내 메시지 즉시 전송

### 6.4 필수 설정 검증

- 시작 시 길드별 다음 키 미설정이면 경고 로그 출력
  - `warning_channel_id`
  - `auth_channel_id`
  - `onboarding_channel_id`
  - `auth_completed_role_id`

## 8. 비기능 요구사항

- 모든 권한 판단은 관리자 권한 기반으로 수행
- 모든 시간 계산/저장은 KST 기준
- 슬래시 명령은 DM이 아닌 길드에서만 동작해야 하는 기능을 명확히 제한
- 오류 발생 시 서비스 전체 중단 대신 길드 단위로 격리해 처리 지속
- 시작/종료 안정성 보장
  - 종료 시 스케줄러와 DB 연결을 정리
  - 네트워크/게이트웨이 오류 시 재시도 루프 적용
- 식별은 이름이 아닌 ID 기반으로 처리

## 9. 구현 주의사항 및 현행 차이

- 신규 입장 권한 부여 방식은 기존의 단순 이모지 반응 역할 부여가 아니라, 임베드 기반 초기 설정 완료 후 일괄 권한 부여 흐름으로 정리한다
- 초기 설정은 공개 채널 스팸을 줄이기 위해 공용 안내 임베드 + 사용자별 상호작용 응답 방식으로 구현하는 편이 적합하다
- 자동 인증 스케줄링은 길드별로 일시 중단할 수 있어야 하며, 일시 중단은 스케줄 tick에만 영향을 주고 수동 관리자 테스트 명령까지 막지는 않는 편이 적합하다
- 일반 입장 흐름은 성별 버튼 화면과 게임 버튼 화면의 2단계로 분리하고, 각 단계 선택 시 즉시 다음 단계 또는 완료로 이어지게 구성한다
- 별도의 공통 입장 권한 역할은 사용하지 않고, 남성/여성/지인 역할을 서버 접근 기준 역할로 활용한다
- 완료 상태를 무겁게 영속 저장하기보다, 사용자가 언제든 잘못 선택을 수정하고 다시 진행할 수 있는 흐름이 우선이다
- 지인 입장자는 일반 온보딩 예외 대상으로 두고, 성별/게임 선택 없이 지인 역할만 빠르게 부여하는 정책이 적합하다
- 역할 지급 전에 고정된 닉네임 양식 변경 안내를 가장 먼저 노출해야 운영 혼선을 줄일 수 있다
- 닉네임 안내는 일반 입장 예시와 지인 입장 예시를 분리해서 보여줘야 혼동을 줄일 수 있다
- 안내 임베드는 단계 제목에 가벼운 이모지를 포함해도 되지만, 실제 정보 전달이 우선되도록 과도한 장식은 피하는 편이 적합하다
- 음성 채널 정책은 기존 채널 재사용이 아니라, 트리거 입장 시 항상 새 임시 채널을 생성하는 방식으로 본다
- `voice_create.allow_duplicate` 키는 기본 설정에 존재하지만, 현재 구현은 값과 무관하게 중복 생성 허용(항상 새 채널 생성) 정책으로 동작함
- 강퇴는 경고 역할 부여 여부와 독립적으로, DB 경고 누적값 기준으로 판단함
- 경고 역할은 상태 표시/동기화 목적이며 강퇴의 근거 데이터는 아님
